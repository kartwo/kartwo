## 0. 目标与北极星指标

做一套**极简到非技术商家也能自部署**的跨境电商后端，帮商家把店开起来、接上收款、开始卖货。

**北极星指标（所有取舍为它让路）：** 非技术商家 **30 分钟内**：下载运行 → 配好收款 → 发布首个商品 → 店面带 HTTPS 可访问。

## 1. 交付形态：一套代码、三种形态、层层包含

**只有一套代码库、一个内核。** "开源 / 自部署商业 / SaaS"是同一代码按开关组合的三种**形态**，呈包含关系：

```
            同一个内核（一套代码库，单租户，对部署模式无感知）
                        │
   ① 开源版          ② 自部署商业版          ③ SaaS 版
   内核本身          内核 + 商业模块          内核 + 商业模块 + 控制面
   MIT 免费          商家自跑、付费(省心)      你托管、按月收(全托管)

   开源版 ⊂ 自部署商业版 ⊂ SaaS 版   （后者永远是前者"加东西"）
```

铁律：代码库唯一，改一处三形态全受益；三者包含非平行，绝不另起炉灶；开源版必须自己就能完整开店赚钱，不阉割；生意是两种（自部署/SaaS），代码是一套。

## 双模式纪律
配额/计费/密钥来源/实例身份等"双模式分叉点"，只做**接口 + 默认实现（自部署语义）**；不在内核写 SaaS 专属逻辑、不写控制面。

---

## 2. 核心设计原则
1. 一套代码、三形态、层层包含；内核单租户、对部署模式无感知。
2. **Data-as-a-folder**：一个店 = 二进制 + 数据目录，可整体打包搬走。
3. 配置外置、走 UI；凭证存库不硬编码。
4. 升级安全第一：自动备份 → 幂等迁移 → 失败回滚。
5. Open Core：内核 MIT 免费，赚钱向能力做商业模块/SaaS，不做侵入式 license 锁。
6. 资源克制：默认可跑 1C1G $5 VPS。
7. 支持成本即生死线：自助诊断做进系统。
8. 安全是贯穿全程的硬约束（§14）。
9. 差异化靠"接口 + 默认实现"消化，内核永不感知模式。

## 3. 技术选型

| 层 | 选型 | 理由 |
|---|---|---|
| 核心后端 | Go（单静态二进制） | 零运行时依赖、跨平台、低资源 |
| 默认数据库 | 内嵌 SQLite（WAL） | 零运维、数据=一个文件、可移植 |
| 升级数据库 | PostgreSQL | 规模/并发变大后切 |
| 媒体存储 | 本地 `./data/media`（默认）→ S3/R2（升级项） | 默认可移植；规模化可切 |
| HTTPS | 内嵌 autocert | 填域名自动签发续期 |
| storefront | 内嵌默认主题（v1）→ 进阶独立 Next.js/Astro | SEO + 边缘 |
| admin | Vue3/React SPA，二进制 embed 托管 | 不单独部署前端 |
| 控制面（SaaS） | 独立外壳，v2+/商业版 | 编排多实例，不进内核 |

> Medusa（Node）仅作参考实现，不引入依赖。

## 4. 系统架构

```
                         [ 商家域名 / SaaS 二级域名 ]
                              │  HTTPS（内嵌 autocert / 通配符证书）
                         [ CDN 边缘缓存 ]
              ┌───────────────┴───────────────┐
              │   Go 内核二进制（单租户，模式无感知） │
              │  内嵌反代/TLS + 安全响应头           │
              │  Storefront API │ Admin API         │
              │  电商内核：商品/变体/购物车状态机/订单/  │
              │  库存(防超卖)/优惠券/税务/退款          │
              │  支付路由层（Geo→网关，沙箱）          │
              │  媒体层（上传/处理/StoragePolicy/可插拔后端）│
              │  数据导入层（CSV/Shopify/Woo）        │
              │  诊断/审计/邮件队列/备份恢复            │
              │  数据层抽象（SQLite ⇄ PostgreSQL）     │
              │  内嵌静态：Admin SPA/默认主题/向导      │
              └───────────────┬───────────────┘
                    [ ./data 数据目录 ]
                    ├── shop.db   (业务数据，不存图)
                    ├── media/ ├─ originals/  └─ derived/
                    ├── config.enc (密钥不在此明文，见 §15)
                    └── backups/  (本地+可选异地)

   ──── SaaS 模式额外（v2+/商业版，不进内核）────
   [ 控制面 Provisioner ] 编排很多份内核实例 + 自动发二级域名（见 §13.1）
```

## 5. 媒体 / 图片存储 + StoragePolicy

- 落盘 + DB 存**相对路径**，绝不把图当 BLOB。
- 服务端处理：压缩、WebP/AVIF、响应式多尺寸、**剥 EXIF**；原图 `originals/`、派生 `derived/`。
- **内容哈希命名**，防重名/去重，不直接用上传文件名（防穿越）。
- 可插拔存储后端：Local 默认 → S3/R2 升级项。
- **CDN 前置**：本地为源 + 边缘缓存。
- 上传安全：magic bytes 校验、隔离不可执行、正确 content-type。

**StoragePolicy（可插拔，对应双模式）：** 内核默认=自部署不设总配额；控制面注入=SaaS 按套餐硬配额（定价杠杆）。超额=挡新上传+提示升级，**绝不删数据、绝不弄坏运营中的店**。

**通用护栏：** 磁盘监控 + 优雅降级（将满时停新上传但店面仍可下单，防 SQLite 写满损坏）；单文件护栏（最大体积默认 10–20MB 可配、超大分辨率降采样、每商品最多 N 张）；孤儿清理（删商品回收图，派生图可重生成）；切 S3/R2 后给用量展示 + 可选成本上限。

## 6. 数据层抽象（SQLite / PostgreSQL 双驱动）
- 统一 Repository 接口；底层对 SQLite+PG 双方言支持良好（sqlc/ent/bun，见 §21 待决）。
- **纯 SQL 迁移、禁 AutoMigrate**；幂等可重入。
- 金额整数(分)；外部 ID Hashid(≥8)/UUIDv7，不暴露自增、不用雪花。
- 统一 `created_at/updated_at/deleted_at`（软删）。
- `migrate-to-postgres` 命令：导出中间格式→导入 PG，不丢数据。

## 8. 升级 / 备份 / 迁移
- **升级**：检测版本差异→自动备份→幂等迁移（向后兼容）→失败回滚不进坏状态→成功启动。
- **备份**：本地定时自动备份 + **可选异地目标**（S3/另机/网盘）；后台一键**全量导出** zip；一键恢复。
- **可移植**：复制 data 目录 + 换二进制（+重输主口令）；Go 交叉编译 Win/macOS/Linux/ARM；格式开放无私有 blob。

## 9. 支付路由层

```go
type PaymentProvider interface {
    Name() string
    SupportedRegions() []Region
    CreatePayment(order) (PaymentSession, error)
    Capture(sessionID) error
    Refund(sessionID, amount int64) error   // 分
    VerifyWebhook(req) (Event, error)        // 双校验
}
```
基于 IP 地理 / Accept-Language / 时区 / 币种金额选通道。
**Webhook 双校验（强制）**：验签名(HMAC-SHA256) + 比对订单号与网关 reference ID。
**沙箱开关 / 退款与 chargeback 跟踪。**
v1 内置 Stripe、PayPal；预留本地方式（Mada/Tabby、东南亚钱包、iDEAL/Klarna）。
收款回境（文档外）：Airwallex/连连/PingPong/Wise；系统只需多币种定价+对账导出。

## 10. 数据导入（抢客钩子）
可插拔 Importer，每来源一适配器。1) 通用 CSV（v1）；2) Shopify CSV/API（v1.x）；3) WooCommerce（v1.x）。
**必须做对**：图片抓回本地(SSRF 防护)；**301 自动生成**（命门，否则迁站排名归零）；变体完整保留；客户密码迁不了走重置/邀请；干跑预览+错误报告。
**预期管理**：商品/图片/库存/重定向优先；订单/客户/折扣"尽力而为+主要记录"。

## 13. 复用 / 多租户 / 双交付模式
- **默认单租户内核**：一商家一实例，配置/主题/数据外置，换商家只换这三样不改代码。
- **可选控制面（Provisioner，v2+/商业版）**：为新商家自动开通独立实例（一容器+一库+一子域名）。**不做单库多租户**。
- 内核保留 `multi_tenant` 开关与钩子，默认关闭。

### 13.1 SaaS 二级域名自动化（v2+/商业版）
商家注册即自动获得 `shopname.kartwo.com`，由控制面 + DNS 体系实现，**内核不参与**：
- **通配符 DNS**：配 `*.kartwo.com` → 指向 SaaS 入口，任意子域自动解析，无需逐商家加记录。
- **通配符证书**：`*.kartwo.com` 一张证书覆盖所有二级域名（Cloudflare 免费提供）。
- **程序化开通**：控制面经 **Cloudflare API** 为新商家创建路由/记录，全自动 provision。
- **商家自定义域名（更后期）**：商家想用自己品牌域名指向 SaaS 时，用 **Cloudflare for SaaS** 批量管理客户自定义域名的证书与路由。
- 与自部署的关系：自部署单域名走内嵌 autocert；SaaS 二级域名走控制面 + 通配符证书。两条路互不污染内核。

## 14. 安全（贯穿全程的硬约束）
凭证加密存后台（不硬编码/进日志/进 git，导出 zip 内保持加密）；PCI 最小化（只走托管支付，卡号不落地）；强制 HTTPS + 安全响应头（HSTS/CSP/X-Frame-Options/X-Content-Type-Options）；密码 argon2id/bcrypt + 会话失效轮换 + 登录限流；参数化查询防 SQLi、输出转义防 XSS、写操作防 CSRF、鉴权+对象级权限防 IDOR；上传/导入(SSRF)安全；**CI 安全门禁 govulncheck + 密钥扫描，无高危方可合主干**；敏感操作审计。

## 15. 加密密钥管理（分模式）
- **自部署**：密钥由商家**主口令派生**（或 env/OS keyring），不随数据明文存放；搬店后重输主口令；丢失提供"重置并重填密钥"兜底。数据与钥匙完全归商家。
- **SaaS**：由**平台托管密钥**（不懂技术商家无法自管）。诚实区分话术：自部署=你的数据你掌钥；SaaS=便利换托管。
- 密钥来源做成接口：内核默认=主口令；控制面注入=平台托管。

## 16. 合规边界（随主攻市场启用）
PCI（托管支付）；EU：GDPR / Cookie consent / PSD2-SCA(3DS2) / VAT-IOSS / 无障碍 a11y(EAA 2025 起对电商生效)；US：销售税 nexus。做成可开关模块。

## 20. 商业边界划线
内核（开源/自部署）= 完整能开店赚钱的一切，不阉割；商业/SaaS = 省心+规模（托管运维/备份/升级、多店控制面、托管发信、技术支持、配额计费、二级域名）。卖"省事"与"规模"，不卖"解锁基本功能"。
